# AssayAtlas v0.1.1

Prepared on March 23, 2026 as the next patch release after `v0.1.0`.

## Summary

`v0.1.1` is a stabilization and modernization release. It does not try to add headline end-user analysis features. Instead, it makes the current AssayAtlas shell more release-worthy by strengthening the persisted runtime seam, adding a small runtime API, improving module-level testability, and removing a visible browser-polish issue around favicon loading.

## Highlights

### Persisted runtime moves closer to a real application core

- Workspace state now lives behind a persisted SQLite repository and service layer.
- The runtime applies schema-versioned migrations automatically.
- Project updates and export jobs now use explicit runtime write paths.
- Workspace events are recorded so runtime changes are easier to inspect and reason about.

### More scalable integration surface

- The app now exposes JSON endpoints for workspace reads, project updates, export jobs, and workspace event history.
- The health endpoint reports both runtime database presence and schema version, which makes deploy checks more operationally useful.

### Better testing shape

- Focused runtime and API tests now cover the persisted runtime seam directly.
- Route-level smoke tests still exist, but they are no longer the only line of defense against regressions.

### Browser polish

- Browsers that request `/favicon.ico` now receive a valid icon response instead of generating a 404 in the console or network panel.

## Validation

Run these commands before cutting the tag:

```bash
./.venv/bin/python scripts/build_examples.py
./.venv/bin/python -m unittest discover -s tests -v
```

For a narrower shell check after favicon or route work:

```bash
./.venv/bin/python -m unittest tests.test_app -v
```

## Upgrade notes

- `data/generated/use_cases.json` remains the tutorial and initial seed source.
- `data/assayatlas.db` is still runtime state and should not be committed.
- If you need the runtime to reseed from a freshly generated manifest, delete `data/assayatlas.db` before starting the app again.
- The runtime schema version expected after this release is `2`.

## Pull-based deploy

```bash
cd /srv/assayatlas
git fetch --tags origin
git checkout main
git pull --ff-only origin main
python3 -m venv .venv
./.venv/bin/pip install -r requirements.txt
./.venv/bin/python scripts/build_examples.py
./.venv/bin/python run.py
```

## Rollback

```bash
cd /srv/assayatlas
git fetch --tags origin
git checkout v0.1.0
./.venv/bin/python scripts/build_examples.py
./.venv/bin/python run.py
```

## Known limitations

- AssayAtlas is still a single shared workspace, not a true multi-user platform.
- The feature builders still live in the large generator module and should be split further in a later modernization slice.
- Export work is still synchronous rather than delegated to a background job worker.
